# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Prime directives

- data corruption is not allowed ever
- data races are not allowed ever
- race conditions are not allowed ever
- every feature or implementation plan should take into account data corruption and race risks
- after every feature implemented the code should be audited for data corruption and race risks
- this is not a v1, this is production, no data loss ever

## Start here

- **`docs/ARCHITECTURE.md`** — read this before making non-trivial changes. It covers the three pluggable traits (`BlobStore`, `MetadataStore`, `RepoAuthz`), upload/download/global-dedup flows, the wire surface, storage layout, auth flow, the native `git-bale` filter, the OpenTelemetry observability story (gated on `OTEL_EXPORTER_OTLP_ENDPOINT`), and a "Tricky bits" section calling out the parts of the Bale protocol that are easy to get wrong (hex encoding, `url_range` end-inclusivity, signed-URL Range matching, idempotency, shard magic tag).
- **`docs/BALE_FORGE_PROTOCOL.md`** — the contract a git forge has to implement to support `git-bale`: an HTTPS `/info/bale/authenticate` endpoint, an SSH `git-bale-authenticate` command, and a `POST /check_access` callback consumed by `bale-server-authz-http`. Read before touching the `git-bale` auth resolver or `HttpAuthz`.
- **Scope** — the server (CAS upload/download/global-dedup endpoints, HS256 token auth), the native `git-bale` filter, alternate trait impls, the per-repo scope check on reconstruction, and per-owner + per-repo storage accounting + quotas (the `/v1/usage/{owner}`, `/v1/usage/repo/{owner}/{repo}`, and `/v1/quotas/{owner}` endpoints) are all implemented. See `docs/STATUS.md`.
- **`crates/git-bale/`** — native Git filter (`filter.bale.process`). Pretty-printed JSON pointer with stable key order (`hash`, `file_size`, `sha256`), shared chunk cache (`~/.cache/bale/chunks` by default), per-repo manifest cache (`.git/bale/manifests/`), and a per-repo **pointer cache** (`.git/bale/clean-cache/<blake3>`) that short-circuits `clean` when the same input bytes have been cleaned before — `git status` / `git diff` on a modified bale file would otherwise re-run full CDC + xorb + shard work on every command. **Clean is offline: `git add` chunks the file via `TranslatorConfig::local_config` and writes xorbs/shards to `.git/bale/staging/` instead of POSTing to the server.** `git-bale push-pending` ensures every bale file the push *advertises* reaches — and is registered under — the repo of the remote being pushed to: it does NOT POST staged xorbs/shards verbatim (that bypasses server global dedup), it reconstructs each file to a temp and re-cleans it through an online `FileUploadSession` pointed at the target (xet POSTs only new xorbs but always its per-session shard, and that `/shards` POST is what makes the server `register_files(<target repo>, …)`). Two file classes: **freshly staged** files (sourced from `.git/bale/staging/`) and **already-pushed-elsewhere** files reachable from the pushed ref delta but drained from staging by an earlier push to a *different* remote (discovered via the same gix rev-walk gc uses, bounded only by the stdin remote shas — NOT by `refs/remotes/<target>/*`, since git refs on a remote don't imply the bale objects were registered there; that false assumption is what broke pushing to a remote already holding the history. Each candidate is probed via `GET /v1/reconstructions/{hash}` with the target token and skipped if already registered; bytes for the rest are re-sourced from whichever remote still holds them — `origin` first — via a read-scoped reconstruct). Auth is minted against the **push target** (`resolver::resolve_for_remote`, fed the pre-push hook's `$2` URL), not `origin`, so an upload lands in the right server-side repo scope — this is what makes multi-remote (two repos on one server) work; without it a clone of the second remote 404s on checkout (`multi-remote` e2e phase). The pre-push hook installed by `git-bale install --local` runs `git-bale push-pending "$@"` (remote name + URL as `$1`/`$2`, ref updates on stdin) before `git push`; run by hand with no args it falls back to `origin` and drains only staging. If no remote can reconstruct an already-committed file, the push fails rather than advertise an unreconstructable pointer. **Clean only ever *adds* to staging; `git-bale gc` (`gc.rs`) reconciles it: it forgets `file-index/` markers whose file hash is no longer referenced by the git index or any unpushed commit (a `rev_walk` over local refs with remote-tracking refs hidden, diffing each commit against its first parent), and sweeps the shared xorbs/shards only once no marker remains (they're deduped across files, so per-file deletion is unsafe). When gc orphans a hash reachable from no local *or* remote ref, it also invalidates that hash's clean-cache entry (`clean_cache::forget_hashes`) so a later re-`git add` does a full clean + re-stage instead of short-circuiting to a pointer with no retrievable backing — without this, `git add` → `restore --staged` (fires post-checkout gc) → re-add → `git stash`/`pop` lost the staged bytes. Without gc at all, a `git add` you then abandon via `git reset`/`git checkout` would leak staged bytes forever and even get uploaded on the next push.** `install --local` also writes `post-checkout`/`post-commit`/`post-merge` hooks that run `git-bale gc || true` (best-effort: a foreign hook of that name is skipped, not clobbered, unlike the correctness-critical pre-push which errors). Driven end-to-end by the `gc-abandon-checkout`/`gc-keeps-unpushed`/`gc-mixed` phases in `tests/e2e/run.py`. Smudge order: hot path (manifest + chunk cache) → lukewarm path (reconstruct from `.git/bale/staging/` if not yet pushed) → cold path (`/v1/reconstructions/`). The cold path is wrapped in a **no-progress watchdog** (`download_cold` in `filter_process.rs`, shared with `git-bale mount`): it counts bytes actually written and aborts with a user-facing error if none arrive for `BALE_DOWNLOAD_STALL_SECS` (default 30s; `0` disables). This exists because reconstruction bytes are fetched by the *client* directly from presigned blob-store URLs, so a blob store reachable by the server but not the client (e.g. a bundled MinIO presigned as the compose-internal `minio:9000`) fails only here — and xet-core's retry layer misclassifies the "host not found" DNS error as retryable, then `ExponentialBackoff::from_millis(3000)` (tokio-retry squares each step) parks the 2nd retry for up to ~2.5h, hanging `git checkout` silently for hours without the watchdog. Auto-resolves `server_url` + token from `origin` via the forge protocol (HTTPS Basic via `git credential fill`, or SSH `git-bale-authenticate`); a **read/download** resolve over HTTPS tries the forge anonymously first and only falls back to `git credential fill` on 401/403, so cloning a **public** repo never prompts (`authenticate_http_with_fallback` in `resolver.rs`, exercised by the `anon-public-clone` e2e phase). Falls back to `bale.serverUrl` + `bale.token` from local git config when both are set. Depends on `xet-data`, `xet-client`, `xet-core-structures`, `xet-runtime` from crates.io, pinned to `=1.5.2`. Uses `gix` for pure-Rust git ops: rev-parse, tree diff, ODB blob reads, `gix::discover` + config snapshot in `config.rs`, `find_remote("origin")` + `remote_names()` (push-pending's multi-remote re-sourcing) in `remote.rs`, and `git_dir().join("hooks")` (with `core.hooksPath` honored from the snapshot) in `install.rs`. Things still shelling out: `git credential fill` and SSH in `resolver.rs` (forge auth — no library equivalent), and `git config --local/--global/--system` writes in `install.rs` (one-time setup, gix's typed-key write API is more complex than the subprocess for arbitrary user keys). The SSH `git-bale-authenticate` subprocess honors git's own ssh-command config — `GIT_SSH_COMMAND` env, else `core.sshCommand` (same precedence git uses), threaded through `RawConfig.ssh_command` into `ssh_subprocess` which runs a configured command via `/bin/sh` with the auth args appended (bare `ssh` when unset). This matters beyond convenience: ssh expands `~` for its default `UserKnownHostsFile` via `getpwuid`, *not* `$HOME`, so a bare `ssh` would write the server's host keys into the developer's real `~/.ssh/known_hosts` even under the e2e harness's isolated `$HOME` — honoring the harness's `GIT_SSH_COMMAND` (which pins `UserKnownHostsFile`) keeps the handshake from polluting it. The `xet-core/` sibling clone is present only as a source reference for Claude — it's not in the build graph, and the root `Cargo.toml` `exclude`s it so cargo doesn't try to descend into its workspace. **Fully-local (no-server) mode:** `git-bale init-local [--store <path>] [--shared]` sets `bale.local=true` + `bale.localStore` in local git config, installs gc hooks, and (with `--shared`) sets `bale.localShared=true` and registers the repo in `<store>/repos/`; it does NOT install the pre-push hook (nothing to drain). Config keys: `bale.local`/`BALE_LOCAL` (bool), `bale.localStore`/`BALE_LOCAL_STORE` (path, tilde-expanded), `bale.localShared`/`BALE_LOCAL_SHARED` (bool). `crate::store::object_store_root` returns staging in server mode; in local mode it returns the configured store (default per-repo `.git/bale/store`, or `~/bale-local` with `--shared`). Per-repo bookkeeping — `file-index/` markers, clean-cache, manifests — always stays under `.git/bale/` even when the object store is shared. Smudge: hot → store-only reconstruct → **hard error** (no staging path, no network, no auth). `push-pending` is a no-op in local mode. gc hides no remote refs (all refs are live; no server as fallback); on shared stores gc is append-only — `git-bale prune --shared` reclaims by rebuilding the store from live-file union across all registered repos, aborting on missing repos unless `--force`. **Store lock (`crate::store::StoreLock`, fs4 advisory `flock`/`LockFileEx`):** every store *mutator* takes one exclusive lock on a sibling lockfile `<store-parent>/.<storename>.lock` (outside the dir so prune's rename can't move it) — `clean` BRIEFLY **per file** across {write objects → write marker} (NOT the filter-process lifetime: a long hold deadlocks when a git porcelain op like `stash`/`commit -a` nests another clean while the outer filter holds the lock — hung the `offline-no-network` phase), `gc` via non-blocking `try_exclusive`, `push-pending`/`prune` across their sweeps/run. Reads (smudge) and the clean-cache-hit path never lock; different stores never contend; OS releases on crash. This closes the object-vs-marker window. The separate **marker-vs-index** window (git updates its on-disk index *after* the filter returns, so a marker can exist before git references it; a racing gc would read the stale index and sweep the new file's only copy) is closed by **gc deferring while any git index/ref `*.lock` exists** (`gc::git_op_in_flight`: `index.lock` for add/commit, `index.<temp>.lock`+`refs/stash.lock` for stash — stash uses a temp index so the main `index.lock` is absent; checked *after* the store lock so a clean can't slip in between). The one lock-free residual (a `git stash`'s commit-write between its temp-index unlock and `refs/stash.lock`) is closed by a **reclaim grace**: gc won't drop/sweep a marker whose content was cleaned within `BALE_GC_GRACE_SECS` (default 600s); clean rewrites the marker mtime on every add (hit included, under the lock) so "marker age" == "time since cleaned". Trade-off: abandoned content lingers up to the grace before reclaim. Tested by `local-gc-grace` (young orphan survives gc, reclaims once aged); the harness sets `BALE_GC_GRACE_SECS=0` elsewhere so the abandon/race phases test the lock + in-flight check directly. `push-pending`'s drain is surgical under the lock (remove only uploaded markers; sweep objects only if none remain) so a concurrent add isn't blanket-deleted. Driven by `local-basic`/`local-gc-abandon`/`local-shared-dedup`/`local-prune-shared` plus `local-clean-gc-race` + `local-cache-hit-gc-race` (deterministic data-loss races via `BALE_TEST_CLEAN_MARKER_DELAY_MS` / `BALE_TEST_CLEAN_CACHE_HIT_DELAY_MS`) and `local-concurrent-stress` (natural-timing add/gc/prune contention, also a deadlock check) in `tests/e2e/baleharness/phases/local.py` (g3 group, no server).
- **`tests/e2e/`** — out-of-band full end-to-end harness (`tests/e2e/run.py`) that bundles `baleforgit-server` + `openssh-server` in one podman container and drives the release `git-bale` binary through stage/unstage/stash/pop/commit/push/clone over real SSH (including the `multi-remote` phase: push to one repo, then to a second repo on the same server and clone it; plus a third repo seeded with the git commits but no bale objects — via a neutered hook — to which a new branch is pushed and cloned, proving push-pending registers under the push target's scope even when the target already holds the history). Includes a 35 MB binary with two slight modifications so chunk-level dedup is verifiable against on-disk numbers AND the `/v1/usage/...` API, plus adversarial cases (wrong token, tampered xorb on disk, quota exceeded, and `upload-guards`: direct-HTTP negative tests of the server's upload-time `verify_xorb_body` integrity gate, the signed transfer-URL guards (forged/expired/Range-mismatch), and write-scope enforcement). The **whole** suite also runs against an S3 (MinIO) blob store via `tests/e2e/run.py --backend s3`: a process-global backend (`baleharness/s3backend.py`, mirroring the coverage-mode global) makes `start_container` bring up a per-run MinIO sidecar + network and auto-inject `BALE_S3_*` for every container, each server scoped to its own bucket prefix derived from its `data_root`; `ServerHandle.disk_*_bytes()` answers from `mc ls` so the storage-accounting assertions work unchanged, and `tampered-xorb` corrupts bucket objects instead of on-disk files. The one S3-only scenario — a server↔MinIO connection drop mid-PUT (via a host-side TcpProxy) — is the `failure-s3-conndrop` phase, which self-skips under the fs backend. A small `tests/e2e/s3lib.py` survives with two phases (`phase_s3_basic`, `phase_s3_dedup`) that `--coverage` runs against the instrumented image so `bale-server-storage-s3` shows up in the report; it has no standalone entry point. The **whole** suite likewise runs against a **Postgres `MetadataStore`** via `tests/e2e/run.py --meta postgres`: `baleharness/pgbackend.py` (mirroring `s3backend.py`) brings up a per-run Postgres sidecar on the shared network and makes `start_container` inject `BALE_POSTGRES_URL` for every container, each server scoped to its own database derived from its `data_root` (the metadata analogue of the S3 per-`data_root` bucket prefix). It composes with `--backend s3`; the only SQLite-specific phase, `failure-dbbusy`, self-skips under `--meta postgres`. Not run by `cargo test --workspace`. Read [`tests/e2e/README.md`](tests/e2e/README.md) — especially the **Design** section — before changing the harness, the bundled `Dockerfile`/`entrypoint.sh`, or any wire bit the test depends on (the resolver's scp-style URL parser, the JSON pointer shape, the `BALE_GRANTS` env format, the `/v1/usage` response shape, the `clean-cache` schema). Pure stdlib Python; runs on Linux/macOS/Windows hosts without pip.
- **`crates/git-bale/src/mount/`** — two read-only FUSE mounts that share infrastructure: `git-bale mount-diff <revA> <revB>` (both sides of a diff, names labelled `name__<label>.ext`) and `git-bale mount <rev>` (single revision, original names). `mount-diff` populates its inode tree up front from the diff result; `mount <rev>` is *lazy* — directories carry an unresolved gix tree OID and read their children only when FUSE first looks inside, so mounting a 100k-file repo doesn't walk 100k tree objects at startup. Plain blobs stream from the ODB (cap 16 MiB), Bale pointers reconstruct via cache-first then network-fallback (same hot/cold path as the smudge filter). The userspace FS driver is dlopened at runtime via `libloading`, never linked, so the workspace builds without it — a `MountBackend` trait under `mount/backend/` selects one impl per target: **`libfuse.rs`** on unix (`libfuse.so.2` on Linux, `libfuse-t.dylib` on macOS — binding is libfuse 2.x because fuse-t advertises 2.9.9 and libfuse 3 silently mis-positions every op slot after `readlink`) and **`winfsp.rs`** on Windows (WinFsp's `winfsp-x64.dll` FUSE-compat layer, entry point `fsp_fuse_main_real`; its `fuse_operations` is a *different* layout from libfuse 2.x — `getdir` at slot 1, i64 `fuse_off_t`, WinFsp's own `fuse_stat`/`fuse_file_info`). The two share the backend-agnostic VFS/reader/diff and the path→node helpers (`cstr_to_path`/`lookup_path` in `backend/mod.rs`); `mount/mod.rs` reaches whichever compiled via the `PlatformBackend`/`ensure_available` aliases. A missing driver triggers a friendly install-hint exit before any other work. See `docs/ARCHITECTURE.md` § "`git-bale mount-diff` and `git-bale mount`" for tricky bits (path-based callbacks, OnceLock active-VFS, never-reconstruct `getattr`, lazy-dir population, per-open tempfile handles, the WinFsp ABI).

## Common commands

```bash
cargo build --workspace
cargo fmt --all
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features -- -D warnings

# Behavior is verified end-to-end only — there are no Rust tests.
python3 tests/e2e/run.py               # full SSH push/clone/stash/gc harness (needs podman)
python3 tests/e2e/run.py --backend s3  # the same whole suite against an S3/MinIO blob store
python3 tests/e2e/run.py --meta postgres # the same whole suite against a Postgres metadata store
```

The pre-commit hook (`.pre-commit-config.yaml`) runs `cargo fmt --check` and the clippy command above with `-D warnings`. **Both must pass before declaring a code task done** — run them yourself, don't wait for the user's commit to find out.

Running the binary needs two hex secrets and either `BALE_GRANTS` (static dev authz) or `BALE_AUTHZ_HTTP_URL` (delegate to an upstream `/check_access`); see `README.md` "Quick start".

## Tests

There are **no Rust unit or integration tests** — behavior is verified purely from the user's perspective by the out-of-band end-to-end harness under `tests/e2e/`. `cargo test --workspace` builds clean but runs nothing. Prove behavior changes by extending the e2e harness, not by adding `#[cfg(test)]` modules.

- `tests/e2e/run.py` — full harness: bundles `baleforgit-server` + `openssh-server` in one podman container and drives the release `git-bale` binary through stage/unstage/stash/pop/commit/push/clone over real SSH, plus chunk-level dedup assertions and adversarial cases (wrong token, tampered xorb, quota exceeded, `upload-guards`). `--backend s3` runs the same whole suite against a MinIO sidecar instead of the fs blob store; `--meta postgres` runs it against a Postgres sidecar instead of the SQLite `meta.db` (the two compose).
- `tests/e2e/baleharness/phases/local.py` — eight no-server phases (group g3): `local-basic` (init-local → add/commit/checkout with no server), `local-gc-abandon` (staged then abandoned change is swept by gc), `local-shared-dedup` (two repos sharing `~/bale-local` dedup at the object level), `local-prune-shared` (`git-bale prune --shared` reclaims unreachable objects while preserving live ones), `local-clean-gc-race` (deterministic clean-vs-gc data-loss race, widened via `BALE_TEST_CLEAN_MARKER_DELAY_MS`), `local-cache-hit-gc-race` (same race via a clean-cache-HIT re-add, widened via `BALE_TEST_CLEAN_CACHE_HIT_DELAY_MS`), `local-concurrent-stress` (per-repo add+gc and shared add+prune under natural concurrency — no hang, no lost file), `local-gc-grace` (reclaim grace protects a young orphan marker and reclaims it once aged, via `BALE_GC_GRACE_SECS`).
- `tests/e2e/s3lib.py` — two S3 phases (`phase_s3_basic`, `phase_s3_dedup`) that `--coverage` runs against the instrumented image; no standalone entry point (the whole suite on S3 is `run.py --backend s3`, and the server↔MinIO drop is the `failure-s3-conndrop` phase).
- `tests/e2e/run_matrix.py` — drives multiple harness configurations (`--s3` runs the whole suite on the S3 backend; `--postgres` on the Postgres metadata store; both flags compose).

Read [`tests/e2e/README.md`](tests/e2e/README.md) — especially the **Design** section — before changing the harness, the bundled `Dockerfile`/`entrypoint.sh`, or any wire bit the harness depends on (the resolver's scp-style URL parser, the JSON pointer shape, the `BALE_GRANTS` env format, the `/v1/usage` response shape, the `clean-cache` schema). Pure stdlib Python; runs on Linux/macOS/Windows hosts without pip.

## Engineering guidelines

- **Comments earn their place.** Prefer self-explanatory code. Only add a comment for the non-obvious *why* — a hidden constraint, a workaround, a load-bearing invariant, an interaction with the wire protocol. No paragraphs, no docstrings that restate the function signature, no "this function does X" headers. Recent commits ("Trim comments a bit", "Optimize comments") deliberately deleted a lot of noise; don't reintroduce it.
- **Trim comments on every code change.** After writing or editing code, do a dedicated pass over the diff to simplify, trim, or remove comments and docstrings — *before* reporting the task done, not as a follow-up the user has to request. Drop function-shape docstrings, "what" comments that restate code, and section banners inside function bodies. Keep only *why* comments, and tighten the survivors (one line, name the actual failure mode in concrete terms).
- **Update the docs in the same change.** Any change that touches a trait signature, a wire endpoint, an env var, the storage schema, the auth flow, or a "Tricky bits" item must update `docs/ARCHITECTURE.md`, `docs/BALE_FORGE_PROTOCOL.md`, `README.md`, and/or this file as appropriate. Stale docs are worse than no docs.
- **Treat user data as load-bearing.** This system is a primary store for large binary artifacts — corruption is not recoverable from git history. Be paranoid about:
  - **Consistency** — writes that span the blob store and the metadata store must remain correct under partial failure. Prefer content-addressed idempotency (re-upload returns success) over locks. When persisting on the filesystem, write to `tmp/` then `rename(2)`. Never leave a half-written xorb or shard visible.
  - **Concurrency** — handlers are `async` and the router is shared across requests. Don't assume serialised access. Verify that two concurrent uploads of the same content, two concurrent reconstructions of the same file, or an upload racing a dedup query all converge to a consistent state.
  - **Security** — the per-repo scope check is load-bearing; don't relax it (404, not 403, on cross-repo probes). Signed transfer URLs must enforce the `Range` match and use constant-time signature compare. Never log JWTs, hub bearers, or signing keys. New endpoints default to *inside* the auth middleware; opting out is a deliberate decision that needs justification in the code.
- **Avoid crashes.** No `unwrap` / `expect` / `panic!` / `unreachable!` on inputs that can come from the network, the filesystem, env vars, or another process. Use `?` with a typed error (`CoreError`, `ApiError`, `anyhow::Error` at binary edges). `expect` is only acceptable for invariants the compiler can't prove but the surrounding code does — leave a one-line comment saying why it can't fail. A server crash kills every in-flight request; a returned error doesn't.

## Don't "fix" these without being asked

`Range:`-on-request support on `/v1/reconstructions/*` is intentionally absent — the server always returns the full file's terms. The native `git-bale` smudge handler passes `0..file_size` (not `FileRange::full()`) when calling `xet-data` to avoid tripping the prefetcher; don't "simplify" that range cap without reading the comment in `crates/git-bale/src/filter_process.rs`. The same cap is mirrored in `crates/git-bale/src/mount/reader.rs` for the same reason.

libfuse in `mount/backend/libfuse.rs` (and WinFsp in `mount/backend/winfsp.rs`) is loaded with `libloading` at runtime, not linked at build time. Don't switch either to `extern "C"` linkage or a `-sys`/SDK crate — that would force every developer to install the FS driver (libfuse / the WinFsp SDK) just to build the workspace, even if they never run a mount.
